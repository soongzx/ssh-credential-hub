# SSH Credential Hub - 架构决策记录 (ADR)

本文件记录项目中的关键架构决策及其理由，供团队成员理解和参考。

## ADR-001: Electron 作为桌面应用框架

**状态**: 已接受

**日期**: 2026-04-26

**背景**: 项目需要开发一款 Windows 环境下的 SSH 连接管理工具，需要本地数据库和系统级集成。

**决策**: 使用 Electron 28 作为应用框架。

**理由**:
- 团队熟悉 Web 技术栈（Vue/TypeScript），开发效率高
- 需要本地文件系统访问和进程调用（启动终端），Web 应用无法满足
- 需要跨平台潜力（虽然目前仅 targeting Windows）
- 社区成熟，文档丰富

**替代方案**:
- Tauri：更轻量，但 Rust 学习成本高，生态不如 Electron 成熟
- WPF/WinUI：原生体验好，但技术栈与团队不符
- Flutter Desktop：跨平台能力强，但桌面端生态仍在发展中

**后果**:
- 应用体积较大（包含 Chromium）
- 内存占用相对较高
- 需要处理原生模块（better-sqlite3）的编译问题

## ADR-002: Vue 3 + Composition API

**状态**: 已接受

**日期**: 2026-04-26

**背景**: 需要选择前端框架和编程范式。

**决策**: 使用 Vue 3，并强制使用 Composition API 和 `<script setup>` 语法。

**理由**:
- Vue 3 的 Composition API 提供更好的 TypeScript 支持
- 逻辑复用更灵活（组合式函数替代 mixin）
- 性能优于 Vue 2，包体积更小
- `<script setup>` 语法更简洁，减少样板代码

**替代方案**:
- React + Hooks：生态更丰富，但学习曲线更陡
- Vue 2 + Options API：团队可能有经验，但 TypeScript 支持较差
- Svelte：编译时优化，但生态相对较小

**后果**:
- 需要团队成员学习 Composition API
- 部分 Vue 2 插件不兼容
- 代码风格需要统一规范

## ADR-003: better-sqlite3 作为数据库

**状态**: 已接受

**日期**: 2026-04-26

**背景**: 需要本地持久化存储，支持 1000+ 连接记录，要求快速查询。

**决策**: 使用 better-sqlite3（同步 SQLite）而非异步 SQLite 库。

**理由**:
- 同步 API 更简单，避免 async/await 传染
- 性能优于异步实现（在单线程场景下）
- 支持 WAL 模式，读写并发性能好
- 零配置，单文件数据库，便于备份和迁移

**替代方案**:
- sqlite3（异步）：Node.js 官方推荐，但 API 更复杂
- LevelDB：键值存储，不适合关系型查询
- IndexedDB：浏览器端存储，主进程无法直接使用
- 纯 JSON 文件：简单但无法处理大量数据和复杂查询

**后果**:
- 需要在 Electron 中处理原生模块编译
- 数据库操作阻塞主线程，需要控制事务粒度
- 需要自行实现迁移机制

## ADR-004: 无 ORM，手写 SQL

**状态**: 已接受

**日期**: 2026-04-26

**背景**: 需要操作 SQLite 数据库，考虑是否使用 ORM。

**决策**: 不使用 ORM，手写 SQL 并通过 Repository 模式封装。

**理由**:
- 项目数据模型简单（5 张表），ORM 收益有限
- better-sqlite3 没有官方推荐的 ORM
- 手写 SQL 性能可控，便于优化
- Repository 模式提供足够的抽象，便于测试

**替代方案**:
- TypeORM：功能丰富但体积大，better-sqlite3 支持有限
- Sequelize：成熟但 API 较老，TypeScript 支持一般
- Knex.js：查询构建器，介于 ORM 和原生 SQL 之间

**后果**:
- 需要手动管理表结构变更（迁移脚本）
- 需要自行处理 SQL 注入防护（使用参数化查询）
- 类型安全需要额外维护（接口与表结构同步）

## ADR-005: Node.js crypto 模块加密

**状态**: 已接受

**日期**: 2026-04-26

**背景**: 需要加密存储密码和私钥口令，考虑加密方案。

**决策**: 使用 Node.js 内置 `crypto` 模块，AES-256-GCM 算法。

**理由**:
- 无需额外依赖，减少供应链风险
- AES-256-GCM 提供认证加密，防止篡改
- Node.js crypto 基于 OpenSSL，经过充分验证
- 密钥通过机器特征生成，绑定当前设备

**替代方案**:
- crypto-js：纯 JavaScript 实现，性能较差
- bcrypt/argon2：适合密码哈希，不适合可逆加密
- Windows Credential Manager：平台绑定，无法备份
- 第三方加密库：增加依赖风险

**后果**:
- 密钥丢失后数据无法恢复（需要设计备份机制）
- 机器特征变更（如硬件更换）可能导致密钥失效
- 需要谨慎处理密钥的内存安全

## ADR-006: Pinia 状态管理

**状态**: 已接受

**日期**: 2026-04-26

**背景**: Vue 3 应用需要全局状态管理，选择状态管理方案。

**决策**: 使用 Pinia 作为状态管理库。

**理由**:
- Vue 官方推荐，与 Vue 3 深度集成
- TypeScript 支持优秀，类型推断完善
- API 简洁，学习成本低
- 支持组合式 API 风格

**替代方案**:
- Vuex 4：Vue 官方旧方案，TypeScript 支持较差
- Provide/Inject：适合小型应用，大型应用难以维护
- RxJS + 自定义 Store：过于复杂，团队学习成本高

**后果**:
- 增加一个依赖（但体积较小）
- 需要定义 Store 规范（命名、结构）

## ADR-007: Naive UI 组件库

**状态**: 已提议

**日期**: 2026-04-26

**背景**: 需要选择 Vue 3 组件库，加速 UI 开发。

**决策**: 使用 Naive UI 作为组件库。

**理由**:
- 专为 Vue 3 设计，Composition API 友好
- TypeScript 支持完善
- 组件丰富，包含树形控件、数据表格等复杂组件
- 主题定制灵活

**替代方案**:
- Element Plus：生态更大，但部分组件 Vue 3 适配不够完善
- Ant Design Vue：企业级组件，但体积较大
- Quasar：功能全面但学习成本高
- 自研组件：完全可控但开发周期长

**后果**:
- 需要学习 Naive UI 的组件 API
- 主题需要定制以匹配应用风格
- 如果决策变更，组件替换成本较高

## ADR-008: Vite 构建工具

**状态**: 已接受

**日期**: 2026-04-26

**背景**: 需要选择前端构建工具，与 Electron 集成。

**决策**: 使用 Vite 作为构建工具。

**理由**:
- 启动速度快（ESM 原生支持）
- 热更新（HMR）性能优秀
- 与 Electron 集成方案成熟（vite-plugin-electron）
- 配置简单，插件生态丰富

**替代方案**:
- Webpack：生态最成熟，但配置复杂，构建慢
- Rollup：Vite 底层使用，但需自行配置
- esbuild：极速，但生态不够成熟

**后果**:
- 需要配置 Electron 主进程和渲染进程的双构建
- 开发环境需要处理 CORS 和 IPC 调试

## ADR-009: 外部终端启动方案

**状态**: 已提议

**日期**: 2026-04-26

**背景**: 应用本身不提供 SSH 连接功能，需要调用外部终端。

**决策**: 通过命令行参数或 URL Scheme 启动外部终端，不直接集成 SSH 库。

**理由**:
- 保持应用轻量，不重复造轮子
- 用户可使用自己熟悉的终端工具
- 避免在应用中处理复杂的 SSH 协议和终端模拟
- 支持多种终端的灵活扩展

**替代方案**:
- 集成 ssh2 库：应用内直接连接，但需实现终端模拟
- 集成 xterm.js：在应用内嵌入终端界面
- 仅支持单一终端：限制用户选择

**后果**:
- 密码传递需要特殊处理（剪贴板/临时文件）
- 无法直接获取连接状态反馈
- 需要处理不同终端的启动参数差异

## ADR-010: 单仓库（Monorepo）结构

**状态**: 已接受

**日期**: 2026-04-26

**背景**: 项目包含主进程、渲染进程、共享代码，考虑仓库组织方式。

**决策**: 使用单仓库（Monorepo），通过目录区分模块。

**理由**:
- 项目规模适中，无需多仓库管理
- 主进程和渲染进程版本需要严格同步
- 共享代码（类型定义）便于维护
- 简化构建和发布流程

**替代方案**:
- 多仓库（Polyrepo）：主进程和渲染进程分离
- 使用 pnpm workspace / Turborepo：适合更大规模

**后果**:
- 需要清晰的目录边界（防止渲染进程代码混入主进程）
- Git 历史会混合前后端提交
